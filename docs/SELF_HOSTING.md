## Prerequisites

### Server Requirements

| Resource | Minimum | Recommended |
|----------|---------|-------------|
| CPU | 1 vCPU | 2 vCPU |
| RAM | 1 GB | 2 GB |
| Disk | 10 GB | 20 GB+ (depends on media uploads) |
| OS | Ubuntu 22.04+ / Debian 12+ | Ubuntu 24.04 |

### Domain Name

You need a domain name pointed at your server's IP address. SSL certificates require a real domain — `localhost` will not work for production.

Create an A record in your DNS provider pointing to your server's IPv4 address.

### Software

**Docker deployment:**
- Docker Engine 24+
- Docker Compose v2 (included with modern Docker installations)

**Non-Docker deployment:**
- Rust 1.77+
- Node.js 20+
- PostgreSQL 16+
- nginx

## Option A: Docker Deployment (Recommended)

### 1. Install Docker

```bash
# Install Docker using the official convenience script
curl -fsSL https://get.docker.com | sh

# Add your user to the docker group (log out and back in after)
sudo usermod -aG docker $USER

# Verify installation
docker compose version
```

### 2. Clone the Repository

```bash
cd /opt
sudo git clone https://github.com/tinyboard/tinyboards-rewrite.git tinyboards
cd /opt/tinyboards
sudo chown -R $USER:$USER /opt/tinyboards
```

### 3. Configure tinyboards.hjson

```bash
cp tinyboards.example.hjson tinyboards.hjson
```

Open `tinyboards.hjson` in your editor and set these **required** values:

```hjson
{
  // Your domain (no protocol, no trailing slash)
  hostname: "boards.example.com"

  // Database password — generate a strong one:
  //   openssl rand -base64 24
  database: {
    password: "your_strong_database_password"
  }

  // Password hashing salt — must be long and random:
  //   openssl rand -hex 32
  salt_suffix: "your_salt_suffix_here"

  // Admin account for first run
  setup: {
    admin_username: "admin"
    admin_password: "your_strong_admin_password"
    admin_email: "you@example.com"   // optional
  }
}
```

Review the rest of `tinyboards.hjson` for optional settings (SMTP, storage backends, rate limits). The comments in `tinyboards.example.hjson` document every available option. The defaults are safe for most installations.

After editing, generate the `.env` file that docker-compose needs:

```bash
./configure.sh
```

> **Note:** The `.env` file is auto-generated by `configure.sh` from your `tinyboards.hjson`. Do not edit `.env` directly — your changes will be overwritten the next time `configure.sh` runs.

### 4. Choose a PostgreSQL Config

Three pre-tuned configurations are provided based on your server's RAM:

| Server RAM | Config file |
|------------|-------------|
| 1 GB | `config/postgresql_1gb.conf` |
| 2 GB | `config/postgresql_2gb.conf` |
| 4 GB+ | `config/postgresql_4gb.conf` |

Copy the appropriate file:

```bash
# Example for a 2GB server:
cp config/postgresql_2gb.conf config/postgresql.conf
```

If your server uses HDD storage instead of SSD, edit `config/postgresql.conf` and change `random_page_cost` from `1.1` to `4.0`.

### 5. Configure Nginx

For initial setup (before SSL), use the HTTP-only config:

```bash
cp config/nginx_site_http_only.conf config/nginx_site.conf
```

Edit `config/nginx_site.conf` if you need to change anything (the defaults work for most setups).

### 6. Run the Initial Setup

Start PostgreSQL first, run migrations, then bring up everything:

```bash
# Start PostgreSQL
docker compose up -d postgres

# Wait for PostgreSQL to become healthy
sleep 5

# Run database migrations
./scripts/run_migrations.sh

# Start all services
docker compose up -d
```

Verify all containers are running and healthy:

```bash
docker compose ps
```

You should see 4 services (postgres, backend, frontend, nginx) all showing "healthy" status.

### 7. SSL Setup

Install certbot and obtain a certificate:

```bash
# Install certbot
sudo apt install -y certbot

# Stop nginx temporarily to free port 80
docker compose stop nginx

# Obtain certificate (replace with your domain)
sudo certbot certonly --standalone -d boards.example.com

# Create a symlink or copy certs where docker-compose expects them
mkdir -p certs
sudo cp /etc/letsencrypt/live/boards.example.com/fullchain.pem certs/
sudo cp /etc/letsencrypt/live/boards.example.com/privkey.pem certs/
sudo chown $USER:$USER certs/*.pem
```

Switch to the SSL-enabled nginx config:

```bash
# Copy the SSL site config (replaces the HTTP-only one)
cp config/nginx_site.conf config/nginx_site.conf.bak
cp config/nginx_site.conf.ssl config/nginx_site.conf

# Or if you're starting fresh with SSL:
# The provided nginx_site.conf in config/ already has SSL
```

Edit the SSL config to set your domain:

```bash
# Replace REPLACE_WITH_YOUR_DOMAIN with your actual domain
sed -i 's/REPLACE_WITH_YOUR_DOMAIN/boards.example.com/g' config/nginx_site.conf
```

Update `tinyboards.hjson` to enable TLS:

```hjson
tls_enabled: true
frontend: {
  use_https: true
}
```

Regenerate `.env` and restart:

```bash
./configure.sh && docker compose up -d
```

Set up automatic certificate renewal:

```bash
# Add to crontab (runs twice daily, standard practice)
echo "0 3,15 * * * certbot renew --deploy-hook 'cp /etc/letsencrypt/live/boards.example.com/*.pem /opt/tinyboards/certs/ && docker compose -f /opt/tinyboards/docker-compose.yml restart nginx'" | sudo tee /etc/cron.d/certbot-renew
```

### 8. Verify Everything is Running

```bash
# Check service status
docker compose ps

# Check backend logs
docker compose logs backend

# Test from the command line
curl https://boards.example.com/
```

You should see the TinyBoards frontend. Log in with the admin credentials you set in `tinyboards.hjson`.

## Option B: Non-Docker (Systemd) Deployment

### Running the Install Script

The `install_nondocker.sh` script automates the full setup on Ubuntu/Debian:

```bash
# Clone the repository
cd /opt
sudo git clone https://github.com/tinyboard/tinyboards-rewrite.git tinyboards
cd /opt/tinyboards

# Copy and edit tinyboards.hjson first
cp tinyboards.example.hjson tinyboards.hjson
nano tinyboards.hjson   # Set hostname, database password, salt_suffix, etc.
# For bare metal, also set:
#   database.host: "localhost"       (instead of "postgres")
#   media.media_path: "/opt/tinyboards/media"

# Run the installer
sudo ./scripts/install_nondocker.sh
```

### What the Script Does

1. Installs system packages (build tools, OpenSSL, libpq)
2. Installs PostgreSQL 16 from the official repository
3. Creates the database and user from `.env` settings
4. Detects RAM and applies the appropriate PostgreSQL tuning config
5. Installs the Rust toolchain via rustup
6. Installs Node.js 20 via NodeSource
7. Copies project files to `/opt/tinyboards`
8. Builds the backend binary (`cargo build --release`)
9. Builds the frontend (`npx nuxi build`)
10. Runs database migrations
11. Installs systemd service units
12. Configures nginx with the HTTP-only config
13. Starts all services

### Manual Steps After the Script

1. **Edit `tinyboards.hjson`** at `/opt/tinyboards/tinyboards.hjson` with your production values, then run `./configure.sh`
2. **Set up SSL** using certbot:
   ```bash
   sudo apt install -y certbot python3-certbot-nginx
   sudo certbot --nginx -d boards.example.com
   ```
3. **Switch to the SSL nginx config:**
   ```bash
   sudo cp /opt/tinyboards/config/nginx_site.conf /etc/nginx/conf.d/tinyboards.conf
   # Update upstream addresses for non-Docker
   sudo sed -i 's/server backend:8536/server 127.0.0.1:8536/' /etc/nginx/conf.d/tinyboards.conf
   sudo sed -i 's/server frontend:3000/server 127.0.0.1:3000/' /etc/nginx/conf.d/tinyboards.conf
   sudo sed -i 's/REPLACE_WITH_YOUR_DOMAIN/boards.example.com/' /etc/nginx/conf.d/tinyboards.conf
   sudo nginx -t && sudo systemctl reload nginx
   ```
4. **Restart services:**
   ```bash
   sudo systemctl restart tinyboards-backend tinyboards-frontend
   ```

## Updating TinyBoards

### Docker

```bash
cd /opt/tinyboards

# Pull latest code
git pull origin main

# Rebuild and restart with zero downtime
docker compose build
docker compose up -d --remove-orphans

# Run any new migrations
./scripts/run_migrations.sh
```

### Non-Docker

```bash
cd /opt/tinyboards

# Pull latest code
git pull origin main

# Rebuild backend
cd backend
cargo build --release --locked
cp target/release/tinyboards_server /opt/tinyboards/tinyboards_server
strip /opt/tinyboards/tinyboards_server

# Rebuild frontend
cd ../frontend
npm ci --frozen-lockfile
npx nuxi build

# Run migrations
cd ..
./scripts/run_migrations.sh

# Restart services
sudo systemctl restart tinyboards-backend tinyboards-frontend
```

## Storage Backends

TinyBoards supports multiple storage backends for media uploads via OpenDAL.

### Local Filesystem (Default)

Media files are stored in `/app/media` (Docker) or the path set in `media.media_path` in `tinyboards.hjson`.
This is the simplest option and works well for small to medium instances.

No additional configuration needed — this is the default.

### S3-Compatible Storage

Works with AWS S3, Wasabi, Backblaze B2, Cloudflare R2, and MinIO.

Add a `storage` section to your `tinyboards.hjson`:

```hjson
storage: {
  type: "s3"
  s3: {
    bucket: "your-bucket-name"
    access_key: "your-access-key"
    secret_key: "your-secret-key"
    region: "us-east-1"
  }
}
```

For non-AWS providers, also set the endpoint:

```hjson
storage: {
  type: "s3"
  s3: {
    bucket: "your-bucket-name"
    access_key: "your-access-key"
    secret_key: "your-secret-key"
    region: "us-east-1"
    // Backblaze B2
    endpoint: "https://s3.us-west-002.backblazeb2.com"
    // Cloudflare R2
    // endpoint: "https://your-account-id.r2.cloudflarestorage.com"
    // MinIO (self-hosted)
    // endpoint: "http://minio.internal:9000"
  }
}
```

> **Wasabi users:** The endpoint **must** match the bucket's region. If your bucket is in
> `us-west-1`, use `https://s3.us-west-1.wasabisys.com`. Using the wrong region endpoint
> will result in `403 Forbidden` errors that are difficult to debug.

### Azure Blob Storage

```hjson
storage: {
  type: "azure"
  azure: {
    account_name: "your-storage-account"
    account_key: "your-account-key"
  }
}
```

### Google Cloud Storage

```hjson
storage: {
  type: "gcs"
  gcs: {
    credential_path: "/path/to/service-account.json"
  }
}
```

In Docker, mount the credentials file as a volume in `docker-compose.yml`:

```yaml
backend:
  volumes:
    - ./gcs-credentials.json:/app/gcs-credentials.json:ro
```

And set the path in `tinyboards.hjson`:

```hjson
storage: {
  type: "gcs"
  gcs: {
    credential_path: "/app/gcs-credentials.json"
  }
}
```

## Backup and Restore

### Setting Up Automated Backups

The `backup.sh` script creates compressed database dumps with a 7-day retention policy.

```bash
# Create the backup directory
sudo mkdir -p /var/backups/tinyboards
sudo chown $USER:$USER /var/backups/tinyboards

# Test a manual backup first
./scripts/backup.sh

# Set up daily backups at 2 AM via cron
crontab -e
# Add this line:
# 0 2 * * * /opt/tinyboards/scripts/backup.sh >> /var/log/tinyboards-backup.log 2>&1
```

### Running a Manual Backup

```bash
./scripts/backup.sh
```

Backups are saved to `/var/backups/tinyboards/` as timestamped `.sql.gz` files.

### Restoring from Backup

**Warning:** Restoring drops and recreates the database. All current data will be lost.

```bash
# List available backups
ls -lh /var/backups/tinyboards/

# Restore from a specific backup
./scripts/restore.sh tinyboards_20240615_020000.sql.gz

# The script will ask for confirmation before proceeding.
```

After restoring, restart the backend:

```bash
# Docker
docker compose restart backend

# Non-Docker
sudo systemctl restart tinyboards-backend
```

## Troubleshooting

### Containers Won't Start

Check logs for the failing service:

```bash
docker compose logs postgres
docker compose logs backend
docker compose logs frontend
docker compose logs nginx
```

Common causes:
- **postgres:** `database.password` not set in `tinyboards.hjson`, or config file syntax error
- **backend:** `database` section in `tinyboards.hjson` is wrong, database not migrated, or port 8536 conflict
- **frontend:** npm build failed, check that `npx nuxi build` succeeds locally
- **nginx:** Config syntax error — run `docker compose exec nginx nginx -t`

### Cannot Connect to Database

```bash
# Check if PostgreSQL is running and healthy
docker compose ps postgres

# Test connection from the backend container
docker compose exec backend curl -v telnet://postgres:5432

# Check PostgreSQL logs
docker compose logs postgres
```

Verify that the `database` section in your `tinyboards.hjson` (user, password, and database name) matches what PostgreSQL was initialized with. If you change these after first run, re-run `./configure.sh`, then delete the `postgres_data` volume and re-initialize:

```bash
docker compose down
docker volume rm tinyboards_postgres_data
docker compose up -d postgres
sleep 5
./scripts/run_migrations.sh
docker compose up -d
```

### Media Uploads Not Working

```bash
# Check the media volume exists and is writable
docker compose exec backend ls -la /app/media/

# Check nginx can read the media volume
docker compose exec nginx ls -la /app/media/

# For S3 storage, check backend logs for error details
docker compose logs backend | grep -i "storage\|s3\|upload"
```

For local storage, ensure the `media_data` volume is properly mounted in both `backend` and `nginx` services.

### SSL Certificate Issues

```bash
# Test certificate validity
openssl s_client -connect boards.example.com:443 -servername boards.example.com </dev/null 2>/dev/null | openssl x509 -noout -dates

# Check nginx SSL config
docker compose exec nginx nginx -t

# Verify cert files exist and are readable
ls -la certs/
```

If using Let's Encrypt, make sure your domain's DNS A record points to the server and ports 80/443 are open in your firewall.

### Out of Disk Space

```bash
# Check disk usage
df -h

# Check Docker disk usage
docker system df

# Clean up unused Docker resources
docker system prune -a --volumes
```

Media files can accumulate over time. Consider using an S3-compatible storage backend for large instances, or set up a periodic cleanup job.

### Performance Is Slow

1. **Check PostgreSQL config:** Make sure you copied the right tuning config for your server's RAM (see [Choose a PostgreSQL Config](#4-choose-a-postgresql-config)).

2. **Check resource usage:**
   ```bash
   docker stats
   htop
   ```

3. **Check slow queries:** PostgreSQL logs queries slower than the threshold set in your config. Check with:
   ```bash
   docker compose logs postgres | grep "duration"
   ```

4. **Consider upgrading RAM:** The 1GB config is very conservative. Moving to 2GB significantly improves PostgreSQL performance.

## Configuration Reference

All configuration is in `tinyboards.hjson`. See the comments in `tinyboards.example.hjson` for full documentation of every available option.

The `.env` file is auto-generated by `configure.sh` for docker-compose. Do not edit `.env` directly — any changes will be overwritten the next time you run `./configure.sh`.
